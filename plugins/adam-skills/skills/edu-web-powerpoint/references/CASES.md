# 案例管理系统

教学演示中的案例分两种类型，各有不同的处理方式。

> **关键约束**：案例是**整场教学级别**的决策，不是某一章的附属。
> 哪个做实操、哪些做演示，直接影响整个项目的章节结构和 Phase 3 产出。
> 因此**案例选定是 Checkpoint Plan 中的独立硬节点**：
> agent 列出全部案例 → 给出推荐 → **等用户明确确认** → 才能继续开发。

---

## 案例类型

| 类型 | 标记 | 说明 | 去向 |
|---|---|---|---|
| **演示案例** | `[演示]` | 适合在 web-ppt 中展示的案例 | 做进 web-ppt 对应章节 |
| **实操案例** | `[实操]` | 适合学生动手练习的案例 | 放入 `case-exercise/` 文件夹 |

---

## 案例来源

### 1. 大纲中有案例

从大纲中提取案例，整理到 `outline.md` 末尾「案例清单」。

### 2. 大纲中无案例

agent **必须**根据知识点自动生成 2~4 个案例。生成时考虑：

- 知识点的核心概念需要哪些案例来说明？
- 哪些案例适合学生动手练习？
- 案例难度是否覆盖不同层次？

生成后写入 `outline.md` 的「案例清单」，走正常的选择流程。

---

## 实操案例的选择标准

选择哪个案例做实操练习，按以下优先级判断：

| 优先级 | 标准 | 说明 |
|---|---|---|
| 1 | **可本地运行** | 纯 HTML / Python / JS，不需要服务器、数据库、外部 API |
| 2 | **步骤清晰** | 学生能独立完成，不需要额外指导 |
| 3 | **有明确验证** | 有明确的输入 → 输出，学生能判断对错 |
| 4 | **难度适中** | 5~15 分钟可完成，不要太简单或太难 |
| 5 | **与知识点强相关** | 练习的就是本节课的核心内容 |

**推荐理由示例**：
```
★ 推荐 "冒泡排序可视化" 作为实操练习：
  - 可以纯 HTML + JS 本地运行
  - 步骤清晰：填入数据 → 点击排序 → 观察过程
  - 有明确验证：排序结果是否正确
  - 难度适中：约 10 分钟
  - 直接练习本节核心算法
```

## 案例清单格式

在 `outline.md` 末尾：

```markdown
## 案例清单

| # | 名称 | 类型 | 简述 | 推荐理由 |
|---|---|---|---|---|
| 1 | 冒泡排序可视化 | [实操推荐] | 学生输入数组，观察排序过程 | 可本地运行、步骤清晰、难度适中 |
| 2 | 排序算法对比动画 | [演示] | 展示不同排序算法的速度差异 | 适合课堂展示，不适合学生动手 |
| 3 | 时间复杂度计算器 | [演示] | 输入代码片段，分析时间复杂度 | 概念可视化，加深理解 |

**agent 推荐**：案例 1 作为实操练习。其余做进 web-ppt。
```

## 演示案例的处理

标记为 `[演示]` 的案例：

1. 在 `outline.md` 的对应章节 step 中规划展示内容
2. 在 Phase 2 网页开发时，做进对应章节
3. 可以是动画、交互、数据可视化等形式
4. 必须遵循 CHAPTER-CRAFT.md 的视觉演示要求

## 实操案例的处理

标记为 `[实操]` 的案例：

### 产出物结构

```
case-exercise/
├── README.md          # 操作说明
├── main.html          # 案例入口（或 main.py / index.js）
└── ...                # 案例所需资源
```

### README.md 必须包含

```markdown
# <案例名称>

## 学习目标
- <这个案例让学生练习什么>

## 操作步骤
1. <步骤 1>
2. <步骤 2>
3. ...

## 预期结果
- <做对了应该看到什么>

## 思考题
1. <引导学生深入思考的问题>
2. <可选：第二个问题>
```

### 案例文件质量标准

- **可本地运行**：双击 `main.html` 或运行 `python main.py` 即可
- **自包含**：所有依赖都在文件夹内，或使用 CDN 链接
- **有注释**：关键代码有中文注释，方便学生理解
- **有提示**：在关键步骤有注释或提示文字
- **可修改**：学生能方便地修改参数、数据进行实验

### 案例文件夹命名

- 英文小写 + 连字符
- 与案例清单中的名称对应
- 示例：`case-exercise/bubble-sort-viz/`

## 案例生成 Agent

当大纲中没有案例时，ewp 需要专门的案例生成步骤：

### 生成流程

1. **分析知识点**：这个知识点的核心概念是什么？学生需要掌握什么？
2. **构思案例**：生成 2~4 个案例，覆盖：
   - 1 个基础案例（验证核心概念）
   - 1 个进阶案例（综合应用）
   - 0~2 个扩展案例（拓展思考）
3. **评估可行性**：每个案例是否满足实操标准？
4. **写入清单**：把案例写入 `outline.md` 的「案例清单」
5. **给出推荐**：推荐哪个做实操，说明理由

### 生成原则

- **与知识点强相关**：案例必须直接练习本节课的内容
- **有教学价值**：不是为了有趣而有趣，要能帮助理解概念
- **可实现**：不要设计太复杂、学生无法独立完成的案例
- **有梯度**：案例之间有难度梯度，照顾不同水平的学生

---

## 案例变更

如果用户在 Checkpoint Plan 阶段想换实操案例：

1. 用户指定新的实操案例
2. agent 更新 `outline.md` 的案例清单
3. Phase 3 生成新的实操案例文件夹

如果用户想增加新案例：

1. 用户描述新案例
2. agent 评估是否满足实操标准
3. 写入案例清单
4. 继续正常流程

---

## 常见问题

| 问题 | 处理 |
|---|---|
| 所有案例都不适合做实操 | agent 需要重新设计案例，或调整现有案例使其满足标准 |
| 用户对推荐有异议 | 按用户选择执行，但 agent 可以说明原推荐的理由 |
| 案例太复杂 | 拆分或简化，确保学生能在 15 分钟内完成 |
| 案例太简单 | 增加思考题或扩展任务，提升教学价值 |
